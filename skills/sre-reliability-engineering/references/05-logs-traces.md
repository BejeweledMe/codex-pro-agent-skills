# 05. Логи и трейсы

## Суть

Метрики отвечают "что произошло", трейсы помогают понять "где в цепочке", а логи объясняют "почему". Эти три вида сигналов не заменяют друг друга. Надёжная observability-система связывает их через общий trace/request ID, чтобы дежурный мог перейти от burn-rate графика к конкретному trace и затем к логам проблемного сервиса.

Главная идея: логи и трейсы должны быть спроектированы заранее. Во время инцидента поздно добавлять trace_id, структурировать сообщения или искать stack trace, которого нет.

## Практики

- Минимум для логирования:
  - сквозной trace/request ID для каждого запроса;
  - лог начала и завершения запроса;
  - trace ID во всех логах сервисов цепочки;
  - stack trace для 5xx и исключений;
  - структурированный JSON для машинной обработки.
- Используйте OpenTelemetry как стандарт инструментирования для метрик, логов и трейсов.
- Настройте context propagation между сервисами, очередями и фоновой обработкой.
- На высоких нагрузках используйте tail sampling:
  - сохранять 100% ошибок;
  - сохранять медленные запросы;
  - семплировать успешные быстрые запросы.
- Используйте exemplars, чтобы на графике Prometheus/Grafana можно было перейти к конкретному trace.
- Храните trace_id в логах, а не только в access log. Иначе цепочка распадается.
- Для небольшого масштаба допустимы простые инструменты, но рост требует централизованного поиска:
  - малый масштаб: grep/SSH может быть временным решением;
  - средний: Loki;
  - большие объёмы: ClickHouse или Elasticsearch;
  - трансформация логов: Vector/Fluent Bit.
- Разделяйте hot и cold retention. Горячее хранение нужно для оперативной диагностики, холодное - для аудита и редких расследований.

Практические ориентиры retention:

| Сигнал | Горячее хранение | Холодное хранение | Зачем |
| --- | --- | --- | --- |
| Метрики | 15-30 дней full resolution | 1-2 года downsampled | тренды, capacity, SLO history |
| Трейсы | 2-7 дней | обычно не хранят | оперативная диагностика инцидентов |
| Логи | 7-14 дней | 30-90 дней в object storage | постмортемы и расследования |
| Ошибки/Sentry | 30-90 дней | зависит от требований | дедупликация и отслеживание regressions |

Audit/compliance retention - отдельное требование. Его нельзя автоматически переносить на operational logs: это быстро делает observability дорогой и менее полезной для дежурных.

Что не логировать:

- пароли, токены, session cookies;
- персональные данные без необходимости и маскирования;
- тела высоконагруженных запросов по умолчанию;
- постоянный DEBUG в production;
- секреты из env, headers и конфигурации.

## Антипаттерны

- Логи в свободном тексте без структурных полей.
- Trace ID есть только в gateway, но не в downstream-сервисах.
- Ошибки пишутся без stack trace или с обрезанным контекстом.
- Head sampling отбрасывает редкие ошибки, которые потом невозможно расследовать.
- Логи настолько подробные, что стоимость хранения заставляет команду выключать нужные события.
- Логи настолько бедные, что во время инцидента остаётся только "что-то упало".
- Нельзя перейти от метрики к trace и от trace к логам.
- Observability-стек не мониторится как production-система.

## Как валидировать

Проверка трассировки:

1. Возьмите один реальный пользовательский запрос.
2. Найдите его trace_id.
3. Проверьте, что trace показывает gateway, сервисы, зависимости и длительности.
4. По trace_id найдите логи каждого сервиса.
5. Убедитесь, что в ошибке есть stack trace, request metadata и безопасный контекст.

Проверка sampling:

- Ошибки и slow traces сохраняются полностью.
- Низкочастотные endpoint не исчезают из выборки.
- Успешные high-volume запросы не перегружают storage.

Проверка безопасности:

- В логах нет токенов, паролей и лишних персональных данных.
- Есть маскирование чувствительных полей.
- Есть retention policy и доступы по ролям.

Признак зрелости: дежурный начинает с SLO/burn-rate графика, кликает в exemplar, видит trace проблемного запроса, затем по trace_id открывает логи конкретного сервиса и получает actionable cause.
