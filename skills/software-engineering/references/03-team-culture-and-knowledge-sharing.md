# Team Culture And Knowledge Sharing

Use when: reviewing collaboration habits, onboarding, ownership, bus factor, team
learning, or whether an agent should ask, document, or involve humans.

## Core Ideas

- Software engineering is a team activity. Isolated genius does not scale because most long-lived software is changed by many people over time.
- Humility, respect, and trust are not soft extras; they are preconditions for honest feedback, early risk discovery, and shared ownership.
- Hiding work delays feedback, raises bus factor, and makes failures more expensive.
- Psychological safety lets engineers ask questions, admit uncertainty, and teach others without status fear.
- Knowledge sharing needs multiple channels: questions, mentorship, docs, office hours, talks, communities, canonical references, and code review.
- Critical knowledge should not live only in one expert's head or in transient chat.
- Readability and standardized mentorship turn code review into a knowledge transfer system, not just a gate.

## Practices

- Share work early enough that feedback can still change direction.
- Ask precise questions with context, what was tried, and what decision is blocked.
- Write down answers that are likely to help the next person.
- Create canonical sources for repeated questions and keep them discoverable.
- Reward teaching, mentoring, and broadening expertise beyond one team.
- Use code review, docs, talks, and examples to make local conventions learnable.
- Reduce bus factor by rotating ownership, pairing, documenting durable procedures, and making systems understandable.

## Anti-Patterns

- Treating hidden code as productivity until the final reveal.
- Rewarding only feature output while ignoring mentoring, documentation, and support work.
- Using questions or mistakes as status attacks.
- Letting chat answers become the only source of truth.
- Keeping one expert as a permanent bottleneck.
- Assuming knowledge sharing has one silver bullet.

## Agent Checklist

- Is the relevant context written down or trapped in a person/chat thread?
- Who else needs to understand this change later?
- Does the proposed work reduce or increase bus factor?
- Should the agent ask a clarifying question, create docs, or point to a canonical source?
- Are review comments teaching reusable principles or just local preference?
- Is feedback early enough to change the design?
